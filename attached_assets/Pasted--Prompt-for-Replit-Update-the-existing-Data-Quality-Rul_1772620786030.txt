Claude must apply all of the following rule generation logic in every analysis:

**Technical Rules (Layer 1 — already exists, keep)**
- Null/completeness checks for mandatory fields
- Data type validation
- Format pattern checks (dates, emails, phone numbers, IDs)
- Length constraints (min/max character counts)
- Allowed values / enumeration checks

**Logical Rules (Layer 2 — new)**
- Range boundaries (age 0–120, percentage 0–100, quantities must be positive, prices must be non-negative)
- Date ordering logic (start_date must be before end_date, created_at before updated_at, birth_date before hire_date)
- Uniqueness constraints (ID fields, reference numbers, codes must be unique)
- Referential integrity (foreign key fields must reference valid parent records)
- Numerical precision rules (financial fields should have exactly 2 decimal places)

**Business Rules (Layer 3 — new)**
- Conditional rules — if field A has value X then field B must not be null (e.g. if payment_status = PAID then payment_date must exist, if contract_type = EXTERNAL then vendor_id must be populated)
- Status transition rules — field values must follow valid state machine logic (e.g. status cannot go from CLOSED back to DRAFT)
- Business classification rules — field values must align with organizational taxonomy or reference data
- Regulatory rules — fields subject to ZATCA VAT regulations, Zakat rules, customs declarations, and e-invoicing (FATOORAH) requirements must meet specific constraints
- SLA rules — date/time fields with business deadlines must be within acceptable windows

**Cross-Field Rules (Layer 3 continued)**
- Detect relationships between fields that imply business constraints
- Flag combinations of fields that together must satisfy a condition
- Identify parent-child field relationships within the same dataset

**Business Logic Warnings (Layer 4 — new)**
- Flag fields with suspicious naming that suggests they might be storing mixed content
- Warn about fields that are likely mandatory but have no constraint defined
- Flag ambiguous field definitions that could cause inconsistent data entry
- Identify fields that probably need a reference/lookup table but don't have one defined
- Warn about fields where the data type inferred from the name conflicts with the data type inferred from sample values
